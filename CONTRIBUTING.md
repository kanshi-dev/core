# Contributing to kanshi-core

Kanshi is built from three repos: [core](https://github.com/kanshi-dev/core), [agent](https://github.com/kanshi-dev/agent), and [dashboard](https://github.com/kanshi-dev/dashboard). Work is planned on the [Kanshi v1.0.0 project board](https://github.com/orgs/kanshi-dev/projects/1); the roadmap and priorities live there. Please pick up (or file) an issue before opening a PR.

## Workflow

1. **Start from an issue.** Use the issue templates; every issue states Overview, why it matters, and checkable acceptance criteria.
2. **Branch from `main`**, named `<type>/<short-description>`, e.g. `fix/ingest-fatalf`, `feat/health-endpoint`.
3. **Commit with [Conventional Commits](https://www.conventionalcommits.org/):** `feat:`, `fix:`, `docs:`, `chore:`, `refactor:`, `test:`, `ci:`. Keep commits scoped and messages in the imperative ("fix ingest crash", not "fixed").
4. **Open a PR** using the PR template and link the issue (`Closes #N`).
5. **CI must pass before merge** (build, vet, test). Do not merge red. Do not push directly to `main`.

## Rules of the repo

- **Bug fixes land with a regression test** that reproduces the bug: it fails before the fix and passes after.
- **Never hand-edit generated files:** everything in `internal/db/*.sql.go` is generated by sqlc. Change the SQL in `queries/`, then run `sqlc generate` from the repo root.
- Standard Go `internal/` layout; no global state; services take `*db.Queries` directly.
- REST responses always use the `{ code, message, data }` envelope via `response.CustomResponse`.

## Development

```bash
docker compose -f ../infra/docker-compose.yaml up -d   # TimescaleDB
go run ./cmd/core                                      # REST :8080, gRPC :50051
go build ./... && go vet ./... && go test ./...        # what CI runs
```

## Versioning

Semver from v1.0.0: bug fixes ship as `v1.0.x` patches, features wait for `v1.1.0`, no breaking API changes within a major version.
